Data Analytics with Prometheus and Grafana

  • Setup
  • Chapter 1 - What is Prometheus? Grafana?
  • Chapter 2 - Examples
  • Resources

This walks through some examples that may be useful for troubleshooting connectivity issues within your cluster.

Setup

You can setup an example cluster by following the Kubernetes Networking Tutorial. When Contiv is installed, it should be installed with Prometheus and Grafana already configured for the cluster.

Chapter 1: What is Prometheus? Grafana?

Prometheus is an open source monitoring platform that is useful for visualizing time series data. You can get monitor and get alerts for your cluster through Prometheus. You can find out more about Prometheus here.

Grafana is also an open source monitoring platform which further enhances the user experience for data visualization. In this tutorial you will learn how Grafana can directly injest data from Prometheus and some useful queries. For more information of Grafana, click here.

There are many other data monitoring platforms besides Prometheus and Grafana, such as Graphite, InfluxDB, OpenTSDB, Nagios, and Sensu. Because Prometheus and Grafana are open source platforms, they allow for easy integration and personalized configuration. To find out more about alternatives, click here.

Let's explore Prometheus and Grafana.

Note: These steps assume you are working with the demo environment and following along with the Kubernetes Networking and Policy tutorials.

Prometheus and Grafana are running as nodeports which are exposed on the host IP. Since they are both configured to run on the master node, let's find out the IP of the master node.

[vagrant@kubeadm-master ~]$ kubectl describe nodes kubeadm-master

We can see that it's IP is 192.168.2.54.

We can also find out what port Prometheus is exposed at.

[vagrant@kubeadm-master ~]$ kubectl get svc -n kube-system
NAME         CLUSTER-IP       EXTERNAL-IP   PORT(S)          AGE
grafana      10.100.180.79    <nodes>       3000:30001/TCP   1h
kube-dns     10.96.0.10       <none>        53/UDP,53/TCP    1h
prometheus   10.111.247.164   <nodes>       9090:30000/TCP   1h

Let's open a browser and navigate to http://192.168.2.54:30000. This will open up the Prometheus UI.

Let's check and make sure that Prometheus is able to fetch data from all the nodes. At the top click on Status and in the drop down menu click on Targets. You should see that there are two targets and that they are UP and green. If it says UNKNOWN it is likely because the environment or prometheus is still configuring.

Go back to Graph from the top tool bar. Click on the drop down menu that says -insert metric at cursor- and pick any metric. Then click on execute and graph right below the execute button.

Now let's move to Grafana.

Grafana is exposed at port 30001 so let's navigate to http://192.168.2.54:30001 in our browser. The Grafana UI will open up and prompt you for a username and password. Username is "admin" and password is "admin". We will be working with Grafana for the remainder of this tutorial because it is feature-rich, easy to use, and very flexible.

After you have logged in, you will see an option to Add data source. Click on that. Fill out the information as show in the following screenshot.

Click on the Grafana logo in the top left of the screen and go to Dashboard and click on new. Click on graph. Click on "Panel Title" and then click on "Edit".

Chapter 2: Examples

Multi-host networking

Now let's go back to our Kubernetes Networking Tutorial and complete chatper 2.

Instead of ping -c 3 10.1.2.3 it maybe more telling to run ping -c 1 10.1.2.3 -s 5000. This tutorial has run this command three times (once every 15 seconds).

We can add the following four queries:

rx_bytes{containerName="contiv-c1"}
rx_bytes{containerName="contiv-c2"}
tx_bytes{containerName="contiv-c1"}
tx_bytes{containerName="contiv-c2"}

We see three steps as expected, so our basic ping works.

Multi-tenancy networking

Finish chapter 3 from the Kubernetes Networking Tutorial.

Again it may be better to send 5000 bytes instead of the default 64 bytes.

Let's add the following queries.

rx_bytes{containerName="contiv-blue-c1"}
rx_bytes{containerName="contiv-blue-c2"}
rx_bytes{containerName="contiv-blue-c3"}
tx_bytes{containerName="contiv-blue-c1"}
tx_bytes{containerName="contiv-blue-c2"}
tx_bytes{containerName="contiv-blue-c3"}

The reason we see that one graph has 6 steps and the other graphs have three steps is because contiv-blue-c1 pings three times to both contiv-blue-c2 and contiv-blue-c3. We first pinged to c2 and then to c3 which is why the graph for c3 is shifted.

External networking
